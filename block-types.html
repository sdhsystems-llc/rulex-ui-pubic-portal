Data Input Blocks

Bring data in
from anywhere

Input blocks define where your rule gets its data. Connect to APIs, databases, static JSON, or read from the rule's own input payload.

INPUT

Input Schema

Declares the typed payload that enters the rule at runtime. Supports JSON Schema with required fields, types, defaults, and nested objects. The foundation of every rule.

{ "loan_amount": number,
  "credit_score": integer,
  "applicant_id": string }
INPUT

Local JSON

Embed static reference data directly into the rule — lookup tables, thresholds, configuration maps. Version-controlled with the rule. No external dependency.

{ "risk_thresholds": {
  "high": 0.75,
  "medium": 0.45 } }
INPUT

External API

Call any REST endpoint and map the response into the rule's data context. Supports auth headers, timeout, retry logic, and response transformation via JSONPath.

GET https://bureau.api/credit/{applicant_id} → credit_report
INPUT

SQL Query

Execute parameterized SQL against your connected database and bind results into the rule context. Supports PostgreSQL, MySQL, MSSQL. Connection pooling built in.

SELECT credit_limit, account_age
FROM customers WHERE id = :applicant_id
Logic Blocks

Express any business logic
with precision

Logic blocks are the core of your rule — they evaluate conditions, apply decisions, and route data based on your business requirements.

LOGIC

Rule Row (IF/THEN)

The fundamental building block. Define conditions (IF) and their consequences (THEN/ELSE). Supports AND/OR/NOT operators, nested groups, and expression functions.

  • 100+ built-in comparison operators
  • Date arithmetic and string functions
  • Nested condition groups with priority
LOGIC

Decision Table

DMN-compliant spreadsheet tables for multi-condition lookups. FIRST, PRIORITY, ALL, and COLLECT hit policies. Coverage analysis ensures no input combination is missed.

  • Import from Excel/CSV
  • Range expressions ([10..50])
  • Multi-output columns
LOGIC

Collection Op

Apply operations across arrays and collections — filter, map, reduce, sum, average, min, max, count, all, any. Chain operations for complex aggregations.

orders.filter(o => o.status == "LATE")
      .sum(o => o.value)
LOGIC

Execute Sub-Rule

Invoke any other published rule and use its output in the current rule's context. Build modular, composable rule hierarchies. Circular dependency detection built in.

  • Pass input, capture output
  • Version-pin sub-rule calls
  • Async execution option

and AI inference

ADVANCED

⏱ Temporal Rule

Model time-sensitive business rules using all 13 of Allen's interval relations — before, after, meets, overlaps, during, starts, finishes and their inverses.

Allen's Relations
before after meets overlaps during starts finishes + 6 inverses

Use case: Flag transactions that occurred during a suspended account window, or detect when a claim overlaps multiple policy periods.

ADVANCED

📊 CEP Rule

Complex Event Processing with sliding, tumbling, and session window semantics. Detect patterns across high-volume event streams in real time.

SLIDING WINDOW
5 failed logins within 2-minute window → trigger lockout
TUMBLING WINDOW
Aggregate transaction volume per 1-hour bucket for daily reporting
SESSION WINDOW
Group user events within an active session (30-min inactivity timeout)

CONFLICT RESOLUTION
Priority · Recency · Specificity · Random
CHAINING MODES

Use case: Clinical decision support where dozens of inter-related medical facts need to be combined to reach a diagnosis or treatment recommendation.

OUTPUT

Output Schema

Defines the typed shape of what the rule returns. Ensures callers always receive a well-formed, predictable response. Maps internal context variables to named output fields with type coercion.

UTILITY

Log Block

Emit structured log entries at any point in the rule execution. Capture intermediate values, add labels, and send to your observability stack. Essential for debugging production rules.
